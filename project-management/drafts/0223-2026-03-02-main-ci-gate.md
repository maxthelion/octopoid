# Main-branch CI gate: block merges when main is red

**Captured:** 2026-03-02

## Raw

> We need a main-CI gate that blocks merge_pr if main is red, preventing cascading failures. check_ci only checks the PR branch, not main. We had 78 consecutive red runs over 4 days with no detection.

## Idea

`check_ci` (draft 185, task 1ea7b68d) checks whether the **PR branch's** CI is green before the gatekeeper can claim. This is necessary but not sufficient. If main is already broken, a PR with green branch CI still merges into broken main, compounding the problem.

The fix: add a `check_main_ci` check that verifies main's latest CI run is green. This runs alongside `check_ci` in the `provisional -> done` transition. If main is red, the task stays in provisional (PENDING) until main is fixed, or bounces to incoming (FAIL) so the implementer can address it.

### Where it fits

```yaml
"provisional -> done":
    checks: [check_ci, check_main_ci]
    on_checks_fail: incoming
    conditions:
      - name: gatekeeper_review
        ...
```

### Implementation

Add `check_main_ci` to `octopoid/checks.py`:

```python
@register_check("check_main_ci")
def check_main_ci(task: dict) -> CheckResult:
    """Block merges when main branch CI is failing."""
    proc = subprocess.run(
        ["gh", "run", "list", "--branch", "main", "--limit", "1",
         "--json", "conclusion"],
        capture_output=True, text=True, timeout=60,
    )
    if proc.returncode != 0:
        return CheckResult.PENDING  # can't determine, wait

    runs = json.loads(proc.stdout)
    if not runs:
        return CheckResult.PASS  # no CI configured

    conclusion = runs[0].get("conclusion", "")
    if conclusion == "success":
        return CheckResult.PASS
    elif conclusion in ("failure", "cancelled"):
        return CheckResult.FAIL
    else:
        return CheckResult.PENDING  # in_progress, queued, etc.
```

When `check_main_ci` returns FAIL, the task bounces to `on_checks_fail: incoming`. This is deliberate — it signals to the implementer (and humans monitoring the queue) that main is broken and needs fixing before new work can land.

### Post-rebase CI wait (optional enhancement)

A further gap: after `rebase_on_base` in the terminal runs, the branch is force-pushed and GitHub CI re-triggers. But `merge_pr` runs immediately without waiting. This means merge-induced regressions (rebase brought in code that breaks the branch's tests) aren't caught.

This is harder to fix because it's in the `runs` pipeline, not the `checks` pipeline. Options:
1. Add a `wait_for_ci` step between `rebase_on_base` and `merge_pr` in the runs list
2. Split the terminal transition: `provisional -> awaiting-merge` (runs rebase, push) → `awaiting-merge -> done` (checks CI again, then merges)

Option 2 is cleaner but requires a new queue state. Option 1 is simpler but couples a polling wait into the step execution.

## Invariants

- **main-ci-gate**: No PR is merged via `merge_pr` while the latest CI run on main has conclusion "failure". Tasks wait in provisional (or bounce to incoming) until main is green.
- **no-cascade-failures**: A single broken merge cannot cause unbounded subsequent broken merges. The system self-corrects by blocking further merges until the break is fixed.

## Context

Postmortem: [CI red on main for 4 days](../postmortems/2026-03-02-ci-red-on-main-4-days.md). PR #252 broke main on Feb 27. 78 consecutive red CI runs. 48+ PRs merged without detection. The existing `check_ci` only checked PR branch CI.

## Open Questions

- Should `check_main_ci` return FAIL (bounce to incoming) or PENDING (hold in provisional)? FAIL is more visible but may cause churn if the fix is already in flight. PENDING silently holds tasks which is less noisy but less visible.
- Should we also add alerting (e.g. pause the system) when main goes red? A scheduler job could check main CI and auto-pause if it detects N consecutive failures.
- Is the post-rebase CI wait worth the added complexity? It catches a narrower class of bug (merge-induced regression) at the cost of slower merge times.
