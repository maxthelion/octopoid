## Summary

This analysis scanned the `octopoid/` package using pytest-cov, vulture, and wily. Coverage data was collected from non-integration tests only (the `orchestrator/` path used by the quality scripts doesn't exist — the package is `octopoid/`). Vulture and wily were run directly from the project root.

7 integration tests are currently **failing**, all sharing a root cause: test assertions expect `task["queue"] == "failed"` but the code now routes first-failure tasks to `requires-intervention` (self-correcting-failure invariant). This is the most urgent finding.

Five additional areas merit attention: `scheduler.py` has a near-zero maintainability index (MI ≈ 0) with cyclomatic complexity of 313 and is 2287 lines; `tasks.py` and `steps.py` also fall in the MI 25–35 danger zone; `jobs.py` has 24% test coverage despite being a core scheduling file; and `queue_utils.py` re-exports 23 symbols that vulture reports as unused.

## Top Recommendations

### 1. Fix 7 failing integration tests — self-correcting-failure invariant [P1, quick]

**Files:** `tests/integration/test_scheduler_mock.py`, `tests/integration/test_git_failure_scenarios.py`, `tests/integration/test_lease_recovery.py`, `tests/integration/test_pid_recovery.py`, `octopoid/tests/test_scheduler_lifecycle.py`

**Evidence:** 7 tests fail with `assert 'requires-intervention' == 'failed'`. The `_handle_fail_outcome` function (result_handler.py:445) intentionally routes first-failure tasks to `requires-intervention` before they can reach `failed`. The tests were written before this invariant was introduced and still assert the old direct-to-failed behavior.

**Action:** Update the failing test assertions to match the new routing: after a first failure, assert `task["queue"] == "requires-intervention"`. For tests that need to verify the "eventually reaches failed" path, chain through the fixer/intervention cycle in the test. Specific tests to fix:
- `test_failed_outcome_moves_to_failed` — mock test hitting real server (also a test isolation issue, 404 on localhost)
- `test_merge_fails_at_merge_time` — expects `failed`, gets `requires-intervention`
- `test_active_lease_not_requeued` — worktree path doesn't exist (rebase_on_base fails on tmp path)
- `test_dead_pid_detected_and_task_moved_to_failed` — expects `failed`, gets `requires-intervention`
- `test_agent_failure_goes_to_failed`, `test_agent_crash_goes_to_failed`, `test_needs_continuation` — same pattern

**Effort:** quick (<1 day) | **Priority: P1**

### 2. scheduler.py — MI ≈ 0, complexity 313, 2287 lines [P2, large]

**Files:** `octopoid/scheduler.py`

**Evidence:** Wily reports MI = 0.077 (effectively zero — the worst possible score). Cyclomatic complexity = 313, making it one of the most complex files in the system. At 2287 lines it's also the largest Python file. Coverage is 51%, leaving 506 lines with no test cover. Vulture flags 2 unused imports (`ensure_worktree`, `_get_fail_target_from_flow`).

**Action:** Split `scheduler.py` into focused modules. The file currently contains: the scheduler loop, agent spawning, PID management, heartbeat logic, lease expiry checks, and result dispatch. Each is a natural extraction candidate. Immediate wins:
- Remove unused imports `ensure_worktree` (line 31) and `_get_fail_target_from_flow` (line 50)
- Extract the `check_and_update_finished_agents` subsystem (~lines 1082–1200) to a `pid_monitor.py`
- Extract the lease-expiry logic (~lines 960–1053) to `lease_monitor.py`
- Extract spawn logic to `agent_spawner.py`

Even a single extraction reduces the complexity score and makes the remaining code easier to cover with tests.

**Effort:** large (>3 days for full split) | **Priority: P2**

### 3. tasks.py + steps.py — MI 25–29, complexity 131/116 [P2, medium]

**Files:** `octopoid/tasks.py` (820 lines, MI 25.5, complexity 131), `octopoid/steps.py` (744 lines, MI 28.5, complexity 116)

**Evidence:** Both files are in the MI 25–50 "high concern" zone by wily's measure. `tasks.py` has 150 uncovered lines (37% miss) including the `create_task_branch`, `push_branch`, and bulk task manipulation paths (lines 201–304, 499–558). `steps.py` has 136 uncovered lines including the entire `rebase_on_base` and `merge_pr` steps (lines 368–551) — these are the steps that are failing in the integration tests above.

**Action:**
- For `tasks.py`: add unit tests for `create_task_branch` and `push_branch` (lines 201–304), which are critical git operations with no coverage. These can be tested with a mock `subprocess.run`.
- For `steps.py`: the `rebase_on_base` step (lines 368–391) requires a real git worktree with a remote. The integration test failures suggest the step isn't being bypassed when the worktree is a mock. Consider a `skip_git_steps` flag or restructure tests to provide a git remote fixture.
- Both files are candidates for further decomposition: `steps.py` mixes git steps, changelog steps, and hook steps.

**Effort:** medium (1–3 days) | **Priority: P2**

### 4. jobs.py — 24% coverage, core scheduling logic [P2, medium]

**Files:** `octopoid/jobs.py` (572 lines, 24% coverage, MI 47.4)

**Evidence:** `jobs.py` handles job spawning, condition evaluation, and periodic scheduling — it's called by the main scheduler loop. Yet 196 of 257 covered statements have no test. The untested sections include `spawn_job` (lines 118–164), `_run_job_instance` (lines 169–188), and all the periodic job scheduling logic (lines 198–242, 264–446). This is compounded by `job_conditions.py` being 0% covered (29 lines total).

**Action:** Add focused unit tests for `spawn_job` and `_run_job_instance` using mock subprocess. Add tests for the job condition evaluation in `job_conditions.py`. These are pure logic functions that don't need real processes. The periodic scheduling logic in `_should_run_job` (lines 264–298) is particularly risky — it controls when jobs fire — and has no test cover at all.

**Effort:** medium (1–3 days) | **Priority: P2**
