# Turn Octopoid into a pip Package

**Status:** Idea
**Captured:** 2026-02-23

## Raw

> Turn octopoid into a pip package. How does that work? Could we have a build step that strips all our project management etc? Are we already doing that?

## Idea

Make octopoid installable via `pip install octopoid` (or from git) so users don't have to clone the repo and manually set up a venv. A proper pip package would make it easier to install, upgrade, and use octopoid as a tool in other projects.

## Current State

We **already have a basic `setup.py`** at the repo root:

```python
setup(
    name="orchestrator",
    version="0.1.0",
    packages=find_packages(),
    install_requires=["pyyaml>=6.0"],
    entry_points={
        "console_scripts": [
            "octopoid=orchestrator.cli:main",
            "orchestrator-scheduler=orchestrator.scheduler:main",
            "orchestrator-init=orchestrator.init:init_orchestrator",
            "orchestrator-migrate=orchestrator.migrate:main",
        ],
    },
)
```

This is functional for `pip install -e .` (editable dev installs), but has issues:
- **Package named `orchestrator`**, not `octopoid` ‚Äî name conflict risk on PyPI
- **No `pyproject.toml`** ‚Äî using legacy `setup.py` only
- **`find_packages()` with no exclude** ‚Äî includes `tests/`, `packages/`, and everything else
- **Missing dependencies** ‚Äî only lists `pyyaml`, but also needs `requests` (SDK), `textual` (dashboard), etc.
- **No build isolation** ‚Äî no MANIFEST.in to control sdist contents
- **Not on PyPI** ‚Äî only installable from local checkout

## What Would Need to Happen

### 1. Migrate to `pyproject.toml` (modern standard)

Replace `setup.py` with a `pyproject.toml` using the standard `[build-system]` and `[project]` tables. This is the modern way to define Python packages.

### 2. Rename package or keep separate identity

Options:
- **Rename package to `octopoid`** ‚Äî rename the `orchestrator/` directory to `octopoid/` and update all imports. Big change but cleaner.
- **Keep `orchestrator` internally, publish as `octopoid`** ‚Äî use `packages` config to map internal name. Confusing.
- **Keep as-is** ‚Äî publish as `orchestrator` (risky name collision on PyPI).

### 3. Exclude non-distributable content

A pip package should NOT include:
- `project-management/` (drafts, postmortems, tasks)
- `tests/` (unless publishing as a separate test package)
- `.octopoid/` (runtime config, agent dirs, flows ‚Äî these are per-project)
- `submodules/` (server code)
- `packages/` (dashboard, server, client ‚Äî separate packages)
- `docs/` (internal development docs)
- `agents/` (if this is project-specific agent config)
- `.claude/` (Claude Code config)
- `CLAUDE.md`, `CLAUDE.local.md`

This is handled by:
- `[tool.setuptools.packages.find]` with `exclude` patterns in `pyproject.toml`
- `MANIFEST.in` for sdist control
- Or using a build backend like `hatch` or `flit` that supports explicit includes

### 4. Separate the "framework" from the "project"

The key architectural question: **what is the pip package?**

Option A: **Just the orchestrator library** ‚Äî `orchestrator/` module with CLI entry points. Users clone the template repo for project-specific config (.octopoid/, agents, flows).

Option B: **Full toolkit** ‚Äî orchestrator + dashboard + SDK + init scaffolding. A single `pip install octopoid` gives you everything.

Option C: **Monorepo with multiple packages** ‚Äî `octopoid` (core), `octopoid-dashboard`, `octopoid-sdk` as separate pip packages. More complex but cleaner separation.

### 5. Build step to strip project management

Yes, this is standard practice. Options:
- **`MANIFEST.in`** with explicit `include`/`exclude` ‚Äî controls what goes into the sdist tarball
- **`pyproject.toml` package discovery** with `exclude` ‚Äî controls what goes into the wheel
- **Build hook** (e.g. hatch build hook) ‚Äî can run arbitrary transforms during build

A simple `MANIFEST.in` would look like:
```
include LICENSE README.md
recursive-include orchestrator *.py *.yaml
recursive-exclude project-management *
recursive-exclude tests *
recursive-exclude submodules *
recursive-exclude packages *
recursive-exclude .octopoid *
```

## Context

The project is currently installed via `pip install -e .` in development. For deployment to other projects, users clone the repo. A proper pip package would simplify adoption and make upgrades as simple as `pip install --upgrade octopoid`.

## Decisions

- **Git-only install** ‚Äî no PyPI, just `pip install git+https://...`
- **Rename `orchestrator/` to `octopoid/`** ‚Äî clean break, avoid name conflicts
- **Full toolkit in one package** ‚Äî core orchestrator + dashboard + SDK all included
- **Server stays separate** ‚Äî separate repo, potentially pulled in as optional for local hosting
- **Include templates as package data** ‚Äî agent dirs, default flows, scaffolding

## Possible Next Steps

- Audit all dependencies that need to be in `install_requires` (pyyaml, requests, textual, etc.)
- Create a `pyproject.toml` with proper package discovery and exclusions
- Decide on package name (octopoid vs orchestrator)
- Add `MANIFEST.in` for sdist control
- Test building with `python -m build` and verify the wheel/sdist contents are clean
- Consider a `[project.optional-dependencies]` section for dashboard (`textual`) vs core
