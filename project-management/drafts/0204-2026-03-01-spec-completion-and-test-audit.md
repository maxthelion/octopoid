# Complete the system spec and audit test coverage against it

**Captured:** 2026-03-01
**Author:** human

## Raw

> It seems like there's 2 things to do. 1. fill up the spec with definitions. This can be based on searching for drafts and tasks and making sensible choices. 2. take inventory of the features that are not well represented in integration tests. So the spec is the source of truth about what we think the system does, and tests etc are artefacts created to check that our expectation of the system matches that description.

## Idea

The system spec (`project-management/system-spec/`) is the source of truth about what the system is supposed to do. Tests are artefacts that verify the system matches that description. Two phases of work:

**Phase 1 — Fill the spec with definitions (multi-pass).** The tree structure exists (15 sections, ~40 invariants from drafts 181/199) but many leaf files have empty `invariants: []` arrays. Fill them using a three-pass approach:

1. **Pass 1 — Explicit invariants from drafts.** For each empty or sparse leaf, use qmd to search drafts for relevant behavioural guarantees. Extract them as invariants. These are the easy ones — someone already wrote them down.

2. **Pass 2 — Best-effort inference.** For leaves still empty after Pass 1, make a best-effort guess at what the invariant should be based on the section/leaf description and general system knowledge. Flag any that are ambiguous or hard to articulate clearly — these may indicate the section itself needs rethinking.

3. **Pass 3 — Implicit invariants from code.** Go through the codebase to find implicit invariants that nobody wrote down but the system relies on. These are the "load-bearing assumptions" — things that would break the system if violated but aren't documented anywhere. Cap at ~10 invariants per section to keep the spec digestible. If that feels too tight, revisit.

Human reviews after each pass before merging into the spec.

**Phase 2 — Audit test coverage against the spec.** Once the spec is populated, walk each invariant and check whether an integration test exists that verifies it. Produce a coverage report: which invariants are `enforced` (have a passing test), which are `aspirational` (no test), and which are testable but untested (the gap to close). This audit becomes the backlog for test-writing tasks.

The relationship is: **spec defines expectations → tests verify expectations → gaps between spec and tests are the work to do.**

## Invariants

- **spec-leaves-populated**: Every leaf YAML file in the system spec tree has at least one invariant. No leaf file has an empty `invariants: []` array.
- **invariant-has-status**: Every invariant in the spec has a `status` field set to either `enforced` or `aspirational`.
- **enforced-has-test**: Every invariant with `status: enforced` has a non-null `test` field pointing to a passing test.
- **coverage-report-exists**: A test coverage audit document exists that maps every spec invariant to its test status (tested, testable-but-untested, architectural/not-directly-testable).

## Open Questions

- Is ~10 invariants per section the right cap, or will some sections naturally need more? (e.g. tasks/resilience already has 9)
- How granular should invariants be? One per feature? One per edge case? The current spec ranges from high-level ("agents are pure functions") to specific ("flow-sync-preserves-steps").
- Should the coverage audit be a one-time document or a living report regenerated by CI/analyst?
- For Pass 3 (implicit invariants from code): should this be a human session or an agent task? Code-reading for implicit assumptions requires strong contextual judgement.
